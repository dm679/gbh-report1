# Отчёт 1: Динамика и конверсия воронки продаж

Локальное приложение для Битрикс24 — аналитический дашборд (только чтение).
SPA во `iframe`, данные через REST API (BX24.js: `callMethod` / `callBatch`),
графики на ECharts. Тяжёлый бэкенд не нужен.

## Структура проекта

```
Отчеты для GBH/
├── index.html                # точка входа SPA (страница приложения в iframe)
├── install.html              # обработчик установки локального приложения (+ placement.bind)
├── css/
│   └── styles.css            # стили, адаптив, печать (PDF)
├── js/
│   ├── config.js             # ★ ЕДИНЫЙ КОНФИГ: UF-поля, пороги, валюта, placement
│   ├── util.js               # даты, бакеты по гранулярности, форматирование
│   ├── bx24.js               # обёртка REST: callMethod/callBatch/fetchAll (+ webhook-режим)
│   ├── data.js               # справочники (кэш), валюты, выгрузка сделок и истории стадий
│   ├── filters.js            # состояние и UI блока фильтров
```

## Блоки отчёта

Вёрстка следует макету (`Скрины/`): фиолетовый баннер-заголовок, фильтры-пилюли
слева, верхняя полоса с жёлтыми карточками, ряд «Динамика | Воронка», таблица
источников внизу.

- **A. Сводные** — Сумма по сделкам (USD) и Средний чек (= сумма / кол-во).
- **B. Сегментация по суммам** — коридоры (пороги из `CONFIG.thresholds`):
  до $100k / $100k–$200k / $200k–$500k / $500k+ (кол-во и сумма). На макете
  нарисовано 3 коридора — по ТЗ оставлены все 4 (включая `$500k+`).
- **Сегменты «ПО ПРОЕКТАМ»** — карточки по разрезу (Проекты / Менеджеры /
  Воронки) с переключателем Сумма/Кол-во (как блок «ПО ПРОЕКТАМ» на макете).
- **C. Динамика** — линейный график: X = время (гранулярность из фильтра),
  Y = две линии (Сумма и Количество). Разрез: Проекты / Менеджеры / Воронки.
- **D. Воронка** — ECharts funnel (широкое основание = входящие). На каждом
  этапе: сумма, конверсия этапа % (по кол-ву и по сумме), тренд ↑/↓/=
  относительно предыдущего аналогичного периода. Источник — `crm.stagehistory.list`.
  Если воронка не выбрана — предупреждение, график не строится.
- **E. Источники** — таблица: Источник | Сумма (USD) | Кол-во | Средний чек.

Фильтры применяются ко всем блокам одновременно.

## Используемые REST-методы

| Данные | Метод |
|---|---|
| Сделки | `crm.deal.list` (пагинация 50 + `callBatch`) |
| Воронки | `crm.category.list` (`entityTypeId=2`) |
| Стадии | `crm.status.list` (`ENTITY_ID=DEAL_STAGE` / `DEAL_STAGE_{CATEGORY_ID}`) |
| Источники | `crm.status.list` (`ENTITY_ID=SOURCE`) |
| Менеджеры | `user.get` (`ACTIVE=Y`) |
| Валюты | `crm.currency.list` + `crm.currency.base.get` |
| История стадий | `crm.stagehistory.list` (`entityTypeId=2`) |
| UF-поля | `crm.deal.userfield.list` |

## Установка

Режим авторизации определяется автоматически (`js/config.js` → `auth.mode`):
на `localhost` — вебхук (разработка), на боевом домене внутри iframe — `bx24`
(OAuth установленного приложения, секрет не нужен).

### Локальная разработка (вебхук)

1. В Битрикс24: **Разработчикам → Другое → Входящий вебхук**, права (scope):
   **CRM** и **Пользователи (user)**. Скопируйте URL.
2. Секрет НЕ хранится в коде. Задайте его один раз в консоли браузера:
   ```js
   localStorage.setItem('rep1:webhook', 'https://<портал>/rest/<ID>/<код>/')
   ```
3. Поднимите статику и откройте `http://localhost:8080/`:
   ```powershell
   node serve.cjs
   ```

### Прод: GitHub Pages + локальное приложение Битрикс24 (OAuth)

1. **Публикация на GitHub Pages** (репозиторий публичный — секрета в коде нет):
   ```powershell
   git init; git add .; git commit -m "Отчёт 1"
   git branch -M main
   git remote add origin https://github.com/<USER>/<REPO>.git
   git push -u origin main
   ```
   Затем в репозитории: **Settings → Pages → Source: Deploy from a branch →
   Branch: `main` / `(root)`**. Через ~1 мин адрес:
   `https://<USER>.github.io/<REPO>/`
2. **Регистрация локального приложения** в Битрикс24:
   **Разработчикам → Другое → Локальное приложение**:
   - **Путь вашего обработчика**: `https://<USER>.github.io/<REPO>/index.html`
   - **Путь для первоначальной установки**: `https://<USER>.github.io/<REPO>/install.html`
   - Права (scope): **crm**, **user** (история стадий входит в `crm`).
   - Отметить «Назначить пункт меню» (по желанию) — приложение появится в левом меню.
3. Установите приложение (кнопка «Установить»). `auth.mode` сам станет `bx24`
   (домен не `localhost`), BX24.js авторизует REST-запросы — вебхук не нужен.
4. Доп. встройка вкладки в раздел CRM (необязательно): включите
   `CONFIG.placement.enabled = true`, задайте `PLACEMENT` (напр. `CRM_DEAL_LIST_MENU`),
   передеплойте — `install.html` вызовет `placement.bind`.

> `.nojekyll` уже добавлен (GitHub Pages отдаёт файлы как есть). `Скрины/`,
> `*.bak`, `shot.png` исключены из репозитория через `.gitignore`.

## Конфиг (js/config.js)

Всё, что зависит от портала, — в одном объекте `CONFIG`:

- `auth` — режим (`bx24`/`webhook`) и URL вебхука для разработки.
- `uf` — **маппинг UF-полей** Направление / Проект / Агентство
  (см. `UF-MAPPING-QUESTIONS.md`). Пока плейсхолдеры не заменены — эти фильтры
  отключены, остальное работает.
- `thresholds` — пороги коридоров `[100000, 200000, 500000]`.
- `targetCurrency` — целевая валюта (`USD`).
- `closeDateOnlyClosed` — при выборе «Дата завершения» фильтровать `CLOSED=Y`.
- `placement` — настройки встройки вкладки.
- `pageSize` / `batchSize` — пагинация (50 / 50).

## Допущения (MVP)

- **Валюта**: суммы приводятся к USD по курсу `crm.currency.list` **на момент
  запроса**. Историческая конвертация по дате сделки в MVP не реализована.
- **Фильтр по сумме** применяется на клиенте после конвертации в USD (в сделках
  `OPPORTUNITY` хранится в разных валютах).
- **Воронка**: «достиг этапа» определяется по наличию записи в
  `crm.stagehistory.list`; сумма этапа — сумма `OPPORTUNITY` (USD) достигших
  сделок из текущего датасета. При выборе нескольких воронок funnel строится по
  первой выбранной.
- **Тренд воронки** сравнивает с предыдущим аналогичным периодом (месяц↔прошлый
  месяц и т.п.), отдельно по количеству и по сумме.
